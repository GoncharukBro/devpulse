# Метрики DevPulse

## Обзор

DevPulse собирает сырые метрики из YouTrack, вычисляет KPI, получает LLM-анализ и отображает результаты.
Цепочка: **YouTrack API → MetricsCollector → KpiCalculator → FormulaScorer → MetricReport (БД) → LLM Worker → Reports API → Фронтенд**.

---

## Сырые метрики (из YouTrack)

### Задачи

| Метрика | Описание | Источник |
|---------|----------|----------|
| `totalIssues` | Общее количество задач за период | YouTrack — задачи по assignee, обновлённые в периоде |
| `completedIssues` | Завершённые задачи (resolved в периоде) | YouTrack — поле `resolved` попадает в `[periodStart, periodEnd]` |
| `inProgressIssues` | В работе (не resolved или resolved вне периода) | YouTrack — все кроме `completedIssues` |
| `overdueIssues` | Просроченные (есть Due Date < сейчас, не resolved) | YouTrack — кастомное поле `Due Date` |
| `issuesWithoutEstimation` | Без оценки (estimation = 0) | YouTrack — кастомное поле `Estimation` |
| `issuesOverEstimation` | С превышением оценки (spent > estimation) | Вычисляемая: сравнение estimation с суммой work items |

### Время (в минутах в БД, часы в API)

| Метрика | Описание |
|---------|----------|
| `totalSpentMinutes` | Общее списанное время за период | YouTrack work items, фильтр по автору и проекту |
| `totalEstimationMinutes` | Общая оценка по всем задачам | YouTrack — кастомное поле `Estimation` |
| `spentByType` | Списания по типам задач (JSON) | YouTrack work items, тип определяется через `taskTypeMapping` |
| `estimationByType` | Оценки по типам задач (JSON) | YouTrack — кастомное поле `Estimation` по типам |

### Процесс

| Метрика | Описание |
|---------|----------|
| `avgCycleTimeHours` | Средний Cycle Time для закрытых задач | YouTrack — история статусов: от первого перехода в `cycleTimeStartStatuses` до последнего перехода в `cycleTimeEndStatuses` |
| `bugsAfterRelease` | Баги, вернувшиеся из релизного статуса | YouTrack — баги (`type=bugfix`), у которых в истории есть переход ИЗ `releaseStatuses` |
| `bugsOnTest` | Возвраты на доработку | YouTrack — закрытые задачи, у которых >1 перехода В `cycleTimeStartStatuses` |


### По типам задач (`issuesByType`)

Типы определяются через `taskTypeMapping` в маппинге полей проекта. Стандартные категории:

- `feature` — новая функциональность
- `bugfix` — исправление ошибок
- `techDebt` — технический долг
- `support` — поддержка
- `documentation` — документация
- `codeReview` — код-ревью
- `other` — прочее (по умолчанию, если тип не замаплен)

### Task Summaries

Для каждой задачи собирается: `id` (idReadable), `summary`, `type`, `spent`. Используется для LLM-классификации задач.

## Вычисляемые KPI

Рассчитываются в `KpiCalculator.calculate()` из сырых метрик.

| KPI | Формула | Единицы | Деление на 0 | Округление |
|-----|---------|---------|--------------|------------|
| **Загрузка** (`utilization`) | `totalSpentMinutes / (40 * 60) × 100` | % | Возвращает 0 | до 1 знака |
| **Точность оценок** (`estimationAccuracy`) | `min(est, spent) / max(est, spent) × 100` | % | est=0 или spent=0 → `null` | до 1 знака |
| **Фокус** (`focus`) | `(feature + techDebt + docs время) / totalSpent × 100` | % | totalSpent=0 → `null` | до 1 знака |
| **Средняя сложность** (`avgComplexityHours`) | `totalSpentMinutes / 60 / completedIssues` | часы | completedIssues=0 → `null` | до 1 знака |
| **Скорость закрытия** (`completionRate`) | `completedIssues / totalIssues × 100` | % | totalIssues=0 → `null` | до 1 знака |
| **Cycle Time** (`avgCycleTimeHours`) | среднее от "в работу" до "закрыто" | часы | нет данных → `null` | до 2 знаков |

### Правила

- Все деления на ноль обработаны (возвращают `null`)
- Загрузка (`utilization`) — единственная метрика, которая при отсутствии данных возвращает 0, а не null
- Значения `null` на фронте отображаются как "Н/Д"
- Все процентные значения не превышают 100% (кроме `utilization`, которая может быть >100% при переработке)

## Score

### LLM Score (0–100)

ИИ-оценка от LLM. Приоритет при отображении.

Промпт включает:
- Данные по задачам (количество, типы, списания)
- Вычисленные KPI
- Список задач для классификации (до 20 шт.)

LLM возвращает JSON с полями: `score`, `summary`, `achievements`, `concerns`, `recommendations`, `taskClassification`.

Парсер (`llm.parser.ts`):
- Пытается `JSON.parse()`, при ошибке — извлекает JSON из текста regex'ом `/{[\s\S]*}/`
- Score кламплен в `[0, 100]` и округлён до целого
- Summary обрезается до 1000 символов
- Массивы фильтруются до строк

### Formula Score (0–100, fallback)

Формульный расчёт (`FormulaScorer`). Используется когда LLM недоступен.

**Веса:**

| Метрика | Вес | Нормализация |
|---------|-----|-------------|
| Загрузка | 20 | Оптимум 80–100%, штраф за >120% |
| Скорость закрытия | 25 | Линейная нормализация 70–100 → 50–100 |
| Точность оценок | 20 | Линейная нормализация 70–100 → 50–100 |
| Фокус | 15 | Линейная нормализация 60–100 → 50–100 |
| Cycle Time | 10 | ≤24ч=100, ≤48ч=80, ≤72ч=60, ≤120ч=40, >120ч=20 |
| Баги (release + test) | 10 | 0=100, 1=80, 2=60, 3–4=40, 5+=20 |

**Общий вес:** 20 + 25 + 20 + 15 + 10 + 10 = **100**

**Правила:**
- Метрики со значением `null` исключаются, их вес перераспределяется пропорционально
- Если ВСЕ метрики `null` → score = `null` (не 0 и не 50)
- Результат кламплен в `[0, 100]` и округлён до целого

### Приоритет отображения

```
llmScore ?? formulaScore ?? null
```

## Цветовая индикация

Определяется в `useMetricColor.ts`. Три уровня: `good` (зелёный), `warning` (жёлтый), `danger` (красный), `neutral` (серый при null).

| Метрика | 🟢 Хорошо | 🟡 Внимание | 🔴 Проблема |
|---------|-----------|-------------|-------------|
| Score | ≥70 | 50–69 | <50 |
| Загрузка | 60–100% | 40–59% или 101–120% | <40% или >120% |
| Точность оценок | ≥75% | 55–74% | <55% |
| Фокус | ≥65% | 45–64% | <45% |
| Скорость закрытия | ≥70% | 50–69% | <50% |
| Cycle Time | ≤48ч | 49–96ч | >96ч |

### Цвета CSS

| Уровень | Текст | Фон | Бордер |
|---------|-------|-----|--------|
| good | `text-emerald-400` | `bg-emerald-500/10` | `border-emerald-500/20` |
| warning | `text-amber-400` | `bg-amber-500/10` | `border-amber-500/20` |
| danger | `text-red-400` | `bg-red-500/10` | `border-red-500/20` |
| neutral | `text-gray-400` | `bg-gray-500/10` | `border-gray-500/20` |

## Concerns (замечания)

Генерируются в `reports.service.ts` при формировании отчётов.

| Условие | Текст | Severity |
|---------|-------|----------|
| `utilization < 50%` | Низкая загрузка (X%) | `warning` |
| `utilization > 110%` | Высокая загрузка (X%) | `danger` |
| `estimationAccuracy < 50%` | Низкая точность оценок (X%) | `warning` |
| LLM concerns непустой | Замечания от LLM | `warning` |
| Score падение > 10 | Падение score: X → Y | `danger` |
| Score падение > 5 | Падение score: X → Y | `warning` |

## Tooltips метрик

Каждая метрика имеет tooltip (компонент `MetricTooltip`) со следующей информацией:

| Метрика | Заголовок | Источник | Формула | Интерпретация |
|---------|-----------|----------|---------|---------------|
| `score` | Оценка продуктивности | LLM-анализ / формульный расчёт | Взвешенная оценка от LLM или формульный расчёт (fallback) | 80–100 — отлично, 60–79 — хорошо, 40–59 — средне, <40 — требует внимания |
| `utilization` | Загрузка | YouTrack (work items) | (Списанное время / 40 часов) x 100% | 60–100% — норма, >100% — переработка, <60% — недозагрузка |
| `estimationAccuracy` | Точность оценок | YouTrack (estimation + work items) | min(оценка, факт) / max(оценка, факт) x 100% | >75% — точные оценки, 55–75% — приемлемо, <55% — нужно улучшить |
| `focus` | Фокус | Вычисляемая (YouTrack) | (feature + techDebt + docs время) / общее время x 100% | >65% — хороший фокус, 45–65% — приемлемо, <45% — много отвлечений |
| `completionRate` | Скорость закрытия | YouTrack (issues) | Завершённые / Всего x 100% | >70% — эффективно, 50–70% — нормально, <50% — много открытых задач |
| `avgCycleTimeHours` | Средний Cycle Time | YouTrack (история статусов) | Среднее время от статуса "В работе" до статуса "Готово" | <48ч — быстро, 48–96ч — нормально, >96ч — медленно |

## Обработка null-значений

| Компонент | Поведение при null |
|-----------|-------------------|
| `KpiCard` | Показывает "Н/Д" серым цветом |
| `ScoreBadge` | Показывает "—" с серым фоном |
| `WeeklyChart` | `connectNulls` — соединяет точки через пропуски |
| Таблицы (EmployeeTable, история) | Используют `formatMetric()` → "Н/Д" |
| TrendIndicator | Показывает "—" серым при отсутствии тренда |

## Форматирование

Определяется в `frontend/src/utils/format.ts`:

| Тип | Формат | Пример |
|-----|--------|--------|
| Проценты | `{value.toFixed(1)}%` (или целое если .0) | `82.3%`, `100%` |
| Часы | `{value}ч` | `36.5ч`, `1.2ч` |
| Score | Целое число | `74`, `91` |
| Даты (короткие) | `dd.mm` | `15.02`, `22.02` |
| Даты (период) | `d–d мес yyyy` | `13–19 янв 2025` |
| null/undefined | `Н/Д` | `Н/Д` |

## Маппинг полей YouTrack

Настраивается per project через сущность `FieldMapping`. Влияет на:

| Маппинг | Описание | Влияет на |
|---------|----------|-----------|
| `taskTypeMapping` | Маппинг YouTrack Type → категория (feature, bugfix, techDebt, ...) | `issuesByType`, `spentByType`, `estimationByType` |
| `cycleTimeStartStatuses` | Статусы начала работы (напр. "In Progress") | `avgCycleTimeHours`, `bugsOnTest` |
| `cycleTimeEndStatuses` | Статусы завершения (напр. "Done", "Closed") | `avgCycleTimeHours` |
| `releaseStatuses` | Статусы релиза (напр. "Released") | `bugsAfterRelease` |

## Период сбора

- **Единица**: неделя (пн 00:00 — вс 23:59 UTC)
- **Автосбор**: по расписанию CRON (настраивается через `CronManager`)
- **Ручной**: любой период через API
- **Backfill**: автоматический поиск и заполнение пропущенных недель

### Дедупликация

Уникальный ключ отчёта: `(subscription, youtrackLogin, periodStart)`. При повторном сборе за тот же период — обновление существующей записи (upsert), не дубль.

## API (Reports)

| Endpoint | Метод | Описание |
|----------|-------|----------|
| `/api/reports/overview` | GET | Общий обзор: средние по всем уникальным сотрудникам, тренд, concerns, weekly trend |
| `/api/reports/employees` | GET | Список всех сотрудников с последним score и трендом |
| `/api/reports/projects/:id/summary` | GET | Сводка по проекту: KPI за последний период, таблица сотрудников, concerns |
| `/api/reports/projects/:id/history` | GET | История проекта по неделям (до 12 недель) |
| `/api/reports/employees/:login/summary` | GET | Сводка по сотруднику: агрегация по проектам, достижения |
| `/api/reports/employees/:login/history` | GET | История сотрудника по неделям (до 12 недель) |
| `/api/reports/employees/:login/report` | GET | Детальный отчёт за конкретный период и проект |
| `/api/reports/employees/:login/reports` | GET | Пагинированный список отчётов |

### Тренды

Рассчитываются по последним 2 значениям score:
- `diff > 5` → `up`
- `diff < -5` → `down`
- иначе → `stable`
- менее 2 значений → `null`

## Страницы фронтенда

### OverviewPage
- Карточки: всего сотрудников, средний score (с трендом), средняя загрузка, тренд
- График: динамика avgScore и avgUtilization по неделям (до 12 недель)
- Concerns: список замечаний по всем сотрудникам
- Последние ачивки

### ProjectPage
- 6 KPI-карточек: Score, Загрузка, Точность, Закрытие, Cycle Time, количество сотрудников
- График с фильтром периодов
- Таблица сотрудников (score, загрузка, точность, закрытие, задачи)
- Concerns по проекту
- LLM-рекомендации

### EmployeePage
- Фильтр по проектам (табы)
- 6 KPI-карточек: Score, Загрузка, Точность, Фокус, Закрытие, Cycle Time
- График с фильтром периодов
- LLM-сводка (summary, достижения, замечания, рекомендации)
- Разбивки по типам задач (2 графика)
- История отчётов (таблица с пагинацией)
- Ачивки
