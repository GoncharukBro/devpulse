## 1. Общая схема процесса

```
Запуск (cron / ручной)
  │
  ▼
Очередь задач (in-memory)
  │
  ▼  ─── Фаза 1: YouTrack ───
  │
  ├─ Для каждого сотрудника × каждой недели:
  │    │
  │    ├─ GET /api/issues        — задачи за период
  │    ├─ GET /api/workItems     — списания за период
  │    ├─ GET /api/issues/{id}/activities — история статусов (для cycle time)
  │    │
  │    ├─ Расчёт сырых метрик (totalIssues, completedIssues, spent, ...)
  │    ├─ Расчёт KPI (utilization, focus, estimationAccuracy, ...)
  │    ├─ Сохранение MetricReport (status='collected')
  │    └─ Генерация ачивок
  │
  ▼  ─── Фаза 2: LLM (асинхронно) ───
  │
  ├─ Очередь LLM-задач
  │    │
  │    ├─ Построение промпта (~700 символов)
  │    ├─ Отправка в LLM (Ollama / внешний API)
  │    ├─ Парсинг JSON-ответа
  │    ├─ Сохранение: score, summary, achievements, ...
  │    └─ MetricReport → status='analyzed'
  │
  ▼
Готово — данные доступны на фронтенде
```

**Ключевой принцип:** YouTrack и LLM — две **отдельные фазы**. Сырые метрики и KPI доступны сразу после фазы 1. LLM-анализ (score, сводка, рекомендации) подтягивается асинхронно и может быть недоступен некоторое время.

## 2. Источники данных из YouTrack

### 2.1 Запрос задач (issues)

**Файл:** `backend/src/modules/collection/metrics-collector.ts`, строка 223

**Запрос:**

```
GET /api/issues?query=project: {PROJECT} assignee: {LOGIN} updated: {START} .. {END}&fields=...
```

**Что означает `updated`:** дата последнего изменения задачи в YouTrack. Обновляется при **любом** изменении:
- Смена статуса
- Добавление комментария
- Списание времени (work item)
- Изменение любого поля (приоритет, тег, оценка, спринт)
- Назначение/переназначение исполнителя

**Получаемые поля:**
- `id`, `idReadable` — идентификаторы задачи
- `summary` — название
- `created` — дата создания (timestamp)
- `resolved` — дата завершения (timestamp, null если не завершена)
- `updatedDate` — дата обновления
- `customFields` — Type, Estimation, Due Date и др.

**Таблица сценариев:**

| Сценарий | Попадёт? | Почему |
|----------|:--------:|--------|
| Задача создана 2 мес. назад, списание 20ч за эту неделю | ✅ Да | Списание обновляет `updated` |
| Задача создана 2 мес. назад, завершена на этой неделе | ✅ Да | Завершение обновляет `updated` |
| Задача создана на этой неделе, в работе | ✅ Да | Создание = обновление |
| Задача создана на этой неделе, уже завершена | ✅ Да | И создание, и завершение обновляют `updated` |
| Задача в backlog, без активности за период | ❌ Нет | Нет обновлений → не попадёт |
| Задача в работе, но на этой неделе без активности | ❌ Нет | Нет обновлений → не попадёт |

**Известное ограничение:** задачи, которые были только прокомментированы (без реальной работы), тоже попадают в выборку. Это ложные попадания (~10-20%), раздувающие `totalIssues`.

**Пагинация:** PAGE_SIZE = 100 записей, автоматическая подгрузка всех страниц.

### 2.2 Запрос списаний (work items)

**Файл:** `backend/src/modules/collection/metrics-collector.ts`, строки 227-242

**Запрос:**

```
GET /api/workItems?startDate={YYYY-MM-DD}&endDate={YYYY-MM-DD}&fields=...
```

Work items получаются **отдельным запросом** с фильтрацией по дате на стороне YouTrack API. Затем на бэкенде фильтруются дополнительно:
- `author.login === employeeLogin` — только списания этого сотрудника
- `issue.idReadable.startsWith(projectShortName + '-')` — только задачи этого проекта

**Получаемые поля:**
- `id` — идентификатор
- `date` — дата списания
- `duration(minutes, presentation)` — длительность
- `type(name)` — тип работы
- `issue(id, idReadable, summary)` — связанная задача
- `author(login, name)` — автор списания

**Важно:**
- Work items привязаны к **дате списания**, а не к дате задачи
- Сотрудник может списать время на чужую задачу (помогал коллеге) — work item попадёт, но задачи в списке issues может не быть
- `totalSpentMinutes` считается **только за период**, а не за всё время жизни задачи

### 2.3 Маппинг полей (Field Mapping)

Маппинг настраивается **per project** (подписка) и определяет:

| Настройка | Описание | Пример |
|-----------|----------|--------|
| `taskTypeMapping` | Значения поля "Type" → внутренние типы | `{"Feature": "feature", "Bug": "bugfix", "Tech Debt": "techDebt"}` |
| `aiSavingWorkType` | Тип work item для ИИ-экономии | `"AI-assisted"` |
| `cycleTimeStartStatuses` | Статусы начала работы (для cycle time) | `["In Progress", "В работе"]` |
| `cycleTimeEndStatuses` | Статусы завершения (для cycle time) | `["Done", "Verified", "Готово"]` |
| `releaseStatuses` | Статусы «релиз» (для метрики баги после релиза) | `["Released", "Deployed"]` |

Если маппинг не настроен или значение не найдено — используется тип `'other'`.

## 3. Сырые метрики

Все метрики, собираемые из YouTrack и сохраняемые в `MetricReport`:

| Метрика | Источник | Описание |
|---------|----------|----------|
| `totalIssues` | issues запрос | Общее кол-во задач, обновлённых за период |
| `completedIssues` | issues с `resolved` в пределах периода | Завершённые за период |
| `inProgressIssues` | issues без `resolved` в пределах периода | Незавершённые (в работе, открытые) |
| `overdueIssues` | issues с просроченным Due Date | Задачи с дедлайном в прошлом и без resolved |
| `issuesByType` | issues + маппинг поля Type | JSON: `{"feature": 5, "bugfix": 2, ...}` |
| `totalSpentMinutes` | work items (сумма `duration.minutes`) | Суммарное списание за период (в минутах) |
| `spentByType` | work items + маппинг типа задачи | JSON: `{"feature": 480, "bugfix": 120, ...}` |
| `totalEstimationMinutes` | issues → поле Estimation | Суммарная оценка задач (полная, не за неделю) |
| `estimationByType` | issues Estimation + маппинг | JSON: `{"feature": 600, ...}` |
| `avgCycleTimeHours` | история статусов завершённых задач | Среднее время от start-статуса до end-статуса (часы) |
| `bugsAfterRelease` | issues Bug + переходы через release-статусы | Баги, обнаруженные после релиза |
| `bugsOnTest` | завершённые issues с возвратами на start-статусы | Задачи, возвращавшиеся на доработку |
| `aiSavingMinutes` | work items с типом aiSavingWorkType | Списание по типу «ИИ-экономия» (в минутах) |
| `issuesWithoutEstimation` | issues без поля Estimation | Задачи без оценки |
| `issuesOverEstimation` | issues где spent > estimation | Задачи с превышением оценки |
| `taskSummaries` | issues (id, summary, type, spent) | Краткие описания задач для LLM |

### Как считаются ключевые метрики

**completedIssues:** задача считается завершённой, если поле `resolved` (timestamp) попадает в диапазон `[periodStart, periodEnd]`. Проверяется в миллисекундах:

```
issue.resolved >= periodStart.getTime() && issue.resolved <= periodEnd.getTime()
```

**inProgressIssues:** все задачи, не попавшие в completedIssues: без resolved, или resolved вне периода.

**totalSpentMinutes:** прямая сумма `duration.minutes` из всех отфильтрованных work items. Это время **только за период**, не за всё время задачи.

**overdueIssues:** задачи с Due Date < текущий момент И `resolved` = null.

**issuesOverEstimation:** задачи где суммарное списание (из work items за период) превышает оценку.

## 4. Вычисляемые KPI

**Файл:** `backend/src/modules/collection/kpi-calculator.ts`

**Общее правило:** если `totalIssues === 0`, **все KPI = null** (нет данных).

Все KPI округляются до одного знака после запятой: `Math.round(value * 10) / 10`.

| KPI | Формула | Когда null | Пример |
|-----|---------|------------|--------|
| **Загрузка** (`utilization`) | `(totalSpentMinutes / 2400) × 100` | `totalIssues = 0` | 1200 мин / 2400 мин = 50.0% |
| **Точность оценок** (`estimationAccuracy`) | `min(estimation, spent) / max(estimation, spent) × 100` | estimation = 0 **или** spent = 0 | min(480, 600) / max(480, 600) = 80.0% |
| **Фокус** (`focus`) | `(feature + techDebt + documentation) / totalSpent × 100` | `totalSpentMinutes = 0` | 900 / 1200 = 75.0% |
| **Средняя сложность** (`avgComplexityHours`) | `totalSpentMinutes / 60 / completedIssues` | `completedIssues = 0` | 1200 мин / 60 / 5 = 4.0 ч |
| **Скорость закрытия** (`completionRate`) | `(completedIssues / totalIssues) × 100`, max 100% | `totalIssues = 0` | 4 / 5 = 80.0% |
| **Cycle Time** (`avgCycleTimeHours`) | среднее(end_timestamp − start_timestamp) в часах | нет завершённых задач или нет переходов | 84.5 ч |

**Пояснения к формулам:**

- **2400 минут** = 40 часов (стандартная рабочая неделя). Загрузка может быть > 100% при переработках.
- **Точность оценок** симметрична: и переоценка, и недооценка снижают показатель. 100% = оценка совпала с фактом.
- **Фокус** считает долю «продуктовых» типов работ. Типы `feature`, `techDebt`, `documentation` считаются целевыми. Всё остальное (bugfix, meetings, other) — нет.
- **Средняя сложность** — сколько часов в среднем тратится на одну завершённую задачу. Больше = задачи сложнее.
- **CompletionRate** ограничена 100% сверху (cap).
- **Cycle Time** — время от первого перехода в start-статус до последнего перехода в end-статус. Рассчитывается только по завершённым за период задачам, на основе истории активностей YouTrack.

## 5. LLM-анализ

### Модель

- Модель настраивается через конфигурацию (по умолчанию: задаётся через env `LLM_MODEL`)
- Temperature: по умолчанию `0.3`
- Rate limit: по умолчанию `3` запроса/минуту (env `LLM_RATE_LIMIT`)

### Что отправляется в LLM

Компактный промпт на **русском языке** (~700 символов), содержащий:

**Системный промпт:** Роль аналитика + формат JSON-ответа + правила оценки score.

**Пользовательский промпт:** агрегированные метрики:

```
Сотрудник: Иванов И.И., проект: MyProject, период: 2026-02-23—2026-03-01

Задачи: 8 всего, 5 закрыто, 1 просрочено
Типы: feature:4, bugfix:2, techDebt:2
Время: списано 32.5ч из 40ч (загрузка 81.3%), оценка 28ч (точность 86.2%)
Фокус: 75%, Completion Rate: 62.5%, Cycle Time: 18.3ч
Баги после релиза: 0, Возвраты: 1
ИИ-экономия: 2.5ч

Задачи (для классификации):
- PROJ-101: Новая форма регистрации (feature)
- PROJ-102: Фикс ошибки авторизации (bugfix)
...
```

**В LLM НЕ отправляются:** исходный код, логи, персональные данные, пароли. Только агрегированные метрики и названия задач (максимум 20 задач).

### Что возвращает LLM

JSON-объект:

| Поле | Тип | Описание |
|------|-----|----------|
| `score` | number (0-100) | Общая оценка продуктивности |
| `summary` | string | Краткая сводка (2-3 предложения) |
| `achievements` | string[] | Ключевые достижения за период |
| `concerns` | string[] | Выявленные проблемы |
| `recommendations` | string[] | Рекомендации по улучшению |
| `taskClassification` | object | Классификация задач: businessCritical, technicallySignificant, bugfixes, other |

### Асинхронность

LLM работает **асинхронно** — после завершения фазы YouTrack отчёты добавляются в LLM-очередь. Фронтенд видит:
1. Сразу — сырые метрики и KPI (MetricReport.status = `'collected'`)
2. Через некоторое время — score и сводку (MetricReport.status = `'analyzed'`)

Интервал между LLM-запросами: `Math.ceil(60000 / rateLimit)` мс. При rateLimit=3 → ~20 секунд между запросами.

## 6. Определение «нет данных»

**Правило:** `totalIssues === 0` означает «нет данных».

| Ситуация | totalIssues | totalSpent | Результат |
|----------|:-----------:|:----------:|-----------|
| Нет задач за период | 0 | 0 | **Нет данных:** все KPI = null, score = null, LLM не вызывается |
| Задачи есть, время не списано | >0 | 0 | **Данные есть:** загрузка = 0%, LLM вызывается |
| Задачи есть, время есть | >0 | >0 | **Данные есть:** все метрики считаются |

### Возможные причины «нет данных»

- **Неправильный маппинг полей проекта** — названия полей YouTrack не совпадают с настройками
- **Сотрудник не назначен на задачи в YouTrack** — фильтр по `assignee` не находит задач
- **Период без активности** — отпуск, болезнь, командировка
- **Задачи не обновлялись в этот период** — все задачи «замерли» на прошлой неделе
- **YouTrack вернул ошибку** — в этом случае MetricReport может не быть создан совсем (зависит от retry)

### Поведение при «нет данных»

1. KPI-калькулятор возвращает null для всех KPI
2. LLM-воркер пропускает отчёт → `llmStatus = 'skipped'`
3. На фронте: score = «Н/Д» (серый бейдж), KPI = «Н/Д»
4. В еженедельном email-отчёте: сотрудник показывается с пометкой «нет данных»

## 7. Score

- **Источник:** исключительно LLM-анализ (formula score убран — поле `formulaScore` deprecated)
- **Диапазон:** 0-100 (целое число)
- **Шкала оценки LLM:**
  - 80-100 — отличная продуктивность
  - 60-79 — хорошая, есть области для улучшения
  - 40-59 — средняя, требуется внимание
  - 0-39 — низкая, серьёзные проблемы

### Когда score = null

1. `totalIssues = 0` → нет данных, LLM не вызывается
2. LLM вернул ошибку → `llmStatus = 'failed'`
3. LLM не смог распарсить ответ → `llmStatus = 'failed'`
4. Сбор был отменён пользователем → `llmStatus = 'skipped'`
5. LLM ещё в очереди → `llmStatus = 'pending'` или `'processing'`

### Отображение на фронте

- Число → цветной круглый бейдж (цвет по порогам, см. секцию 11)
- null → серый бейдж с символом «—», tooltip: «Нет данных» или пользовательская причина

## 8. Статусы и жизненный цикл

### MetricReport.status

| Статус | Значение | Описание |
|--------|----------|----------|
| `collected` | Данные собраны | YouTrack-фаза завершена, KPI рассчитаны |
| `analyzed` | LLM выполнен | LLM-анализ завершён, score и сводка доступны |
| `failed` | Ошибка | Критическая ошибка при сборе данных |

### MetricReport.llmStatus

| Статус | Значение | Описание |
|--------|----------|----------|
| `pending` | В очереди | Отчёт ждёт своей очереди на LLM-анализ |
| `processing` | Обработка | LLM-запрос отправлен, ожидание ответа |
| `completed` | Завершён | LLM-анализ выполнен успешно |
| `failed` | Ошибка | LLM вернул ошибку или невалидный ответ |
| `skipped` | Пропущен | Нет данных (`totalIssues = 0`) или отменён пользователем |

### CollectionLog.status

| Статус | Значение | Описание |
|--------|----------|----------|
| `pending` | В очереди | Задача создана, ожидает обработки |
| `running` | Выполняется | Сбор идёт — обрабатываются сотрудники |
| `stopping` | Останавливается | Получен сигнал остановки, текущий сотрудник дообрабатывается |
| `completed` | Завершён | Все сотрудники обработаны успешно |
| `partial` | Частично | Часть сотрудников завершена с ошибками |
| `stopped` | Остановлен | Остановлен пользователем (текущий дообработан, остальные пропущены) |
| `cancelled` | Отменён | Был в очереди и отменён до начала обработки |
| `failed` | Ошибка | Критическая ошибка (не удалось загрузить подписку, нет сотрудников) |
| `skipped` | Пропущен | Все данные уже актуальны (`overwrite=false`) |

### CollectionLog.type

| Тип | Описание |
|-----|----------|
| `manual` | Запущен пользователем через UI / API |
| `cron` | Запущен автоматически по расписанию |

### Диаграмма переходов CollectionLog

```
pending → running → completed
                  → partial (есть ошибки)
                  → stopped (остановлен)
                  → failed (критическая ошибка)
                  → skipped (всё актуально)
pending → cancelled (отменён до старта)
running → stopping → stopped
```

## 9. Расписание и управление сбором

### Автоматический сбор (cron)

- **Расписание по умолчанию:** `0 0 * * 1` — каждый понедельник в 00:00 (серверное время)
- Настраивается через env `CRON_SCHEDULE` (формат cron: `minute hour * * dayOfWeek`)
- Включение/выключение: env `CRON_ENABLED` (по умолчанию `false`)
- **Что собирает:** данные за **предыдущую неделю** (пн-вс)
- **Для каких проектов:** все активные подписки

### Ручной запуск

| Действие | API endpoint | Описание |
|----------|-------------|----------|
| Запуск для одной подписки | `POST /api/collection/trigger` | Сбор для конкретного проекта |
| Запуск для всех | `POST /api/collection/trigger-all` | Сбор для всех активных подписок |
| Backfill одного проекта | `POST /api/collection/backfill` | Восполнение пропусков за диапазон дат |
| Backfill всех | `POST /api/collection/backfill-all` | Восполнение по всем подпискам |

### Управление

| Действие | API endpoint | Описание |
|----------|-------------|----------|
| Остановить конкретные | `POST /api/collection/stop` | Останавливает указанные подписки |
| Остановить все | `POST /api/collection/stop-all` | Останавливает весь сбор |
| Приостановить cron | `POST /api/collection/cron/pause` | Cron перестаёт срабатывать |
| Возобновить cron | `POST /api/collection/cron/resume` | Cron продолжает работу |
| Состояние cron | `GET /api/collection/cron/state` | `{enabled, schedule, nextRun}` |

### Overwrite (перезапись)

- По умолчанию `overwrite = false` — если MetricReport уже существует и `llmStatus = 'completed'`, сотрудник пропускается
- Если `overwrite = true` — данные пересобираются заново
- Если MetricReport есть, но `llmStatus ≠ 'completed'` — сотрудник ставится в re-queue для LLM (не пересобирается)

### Конфликты

- Нельзя запустить сбор для подписки, которая уже обрабатывается (ответ 409)
- Cron пропускает тик, если уже есть активный сбор

## 10. Очередь и последовательность

### Обработка сотрудников

Сотрудники обрабатываются **последовательно** (один за другим), не параллельно. Порядок:
1. Загрузить подписку, список сотрудников, маппинг полей
2. Определить недели в периоде: `getWeeksBetween(periodStart, periodEnd)`
3. Для каждой недели → для каждого сотрудника:
   - Проверить overwrite / re-queue
   - Собрать метрики из YouTrack (с retry)
   - Рассчитать KPI
   - Сохранить MetricReport
   - Сгенерировать ачивки
4. Добавить собранные отчёты в LLM-очередь

### Retry и задержки

| Параметр | Значение | Описание |
|----------|----------|----------|
| Retry сбора метрик | 3 попытки | На уровне `collectWithRetry` |
| Backoff сбора | 1с, 2с, 4с | Exponential: `1000 × 2^(attempt-1)` мс |
| HTTP retry в YouTrack | 3 попытки | На уровне `youtrack.client` |
| HTTP backoff | 0.5с, 2с, 4с | Exponential: `500 × 2^attempt` мс |
| HTTP timeout | 30 секунд | Таймаут на один запрос |
| Пагинация | 100 записей/страница | Автоматическая подгрузка |
| Поллинг очереди (пустая) | 2 секунды | Интервал проверки новых задач |
| Поллинг очереди (есть задачи) | 100 мс | Между обработкой задач |

### LLM-очередь

| Параметр | Значение | Описание |
|----------|----------|----------|
| Rate limit по умолчанию | 3 запроса/мин | Настраивается через `LLM_RATE_LIMIT` |
| Интервал между запросами | ~20 секунд | `ceil(60000 / rateLimit)` мс |
| Поллинг (пустая очередь) | 3 секунды | `POLL_INTERVAL` |
| Поллинг (есть задачи) | 100 мс | Между обработкой задач |
| Таймаут остановки | 60 секунд | Ожидание завершения текущей LLM-задачи |

### Остановка сбора

При остановке:
1. **YouTrack-фаза:** текущий сотрудник дообрабатывается, остальные пропускаются
2. **LLM-фаза:** задачи в очереди для остановленных подписок помечаются как `'skipped'`
3. CollectionLog получает статус `'stopped'`

### Восстановление после сбоя

При старте сервера:
1. Найти все CollectionLog со статусом `'running'` (не завершились из-за краша)
2. Пометить их как `'failed'`
3. Создать новые логи с `overwrite=true` и добавить в очередь
4. Восстановить незавершённые LLM-задачи: `status='collected' AND llmProcessedAt IS NULL`

## 12. Известные ограничения

### 1. Ложные попадания issues

Фильтр `updated` захватывает задачи, обновлённые без реальной работы (только комментарий, смена приоритета). Это раздувает `totalIssues` на ~10-20% и занижает `completionRate`.

**Планируется:** пост-фильтрация на бэкенде — оставлять только задачи с work items, resolved или created за период.

### 2. Work items на чужие задачи

Если сотрудник списал время на задачу, назначенную на другого (помогал коллеге):
- Work item попадёт (фильтр по `author.login`)
- Но задачи в списке issues не будет (фильтр по `assignee`)
- Тип задачи → `'other'`
- `spentByType` и `focus` будут неточными

### 3. Часовые пояса

- Все периоды (week range) рассчитываются в **UTC**: понедельник 00:00:00 UTC — воскресенье 23:59:59 UTC
- Даты для YouTrack API форматируются как `YYYY-MM-DD` (UTC)
- PostgreSQL хранит даты в UTC
- Node.js pg driver настроен на парсинг `date` как UTC midnight (fix: `pg.types.setTypeParser`)
- **Потенциальная проблема:** сотрудники в часовых поясах, сильно отличающихся от UTC, могут видеть «смещённые» недели

### 4. LLM-зависимость

- При недоступности LLM-сервиса: score = null, сводка отсутствует
- Сырые метрики и KPI доступны **всегда** (не зависят от LLM)
- LLM может вернуть невалидный JSON → `llmStatus = 'failed'`, повтор не делается
- LLM может дать неконсистентный score (разные запуски → разные оценки)

### 5. Эталонная нагрузка

- 2400 минут (40 часов) — фиксированный эталон для загрузки
- Не учитывает: короткие недели, праздники, частичную занятость, отпуска
- Загрузка > 100% просто означает переработку, не ошибку

### 6. Estimation Accuracy

- Работает только если у задачи есть и оценка, и фактическое списание
- Если оценка = 0 или списание = 0 → null (не считается)
- Не учитывает задачи, оценённые в story points (только minutes)

### 7. Cycle Time

- Зависит от корректной настройки `cycleTimeStartStatuses` и `cycleTimeEndStatuses`
- Считается только для задач, завершённых за период
- Если задача была сразу создана в end-статусе (без перехода) → не учитывается
- Использует историю активностей YouTrack, что требует дополнительных API-запросов

## Приложение: Глоссарий

| Термин | Описание |
|--------|----------|
| **MetricReport** | Запись с метриками одного сотрудника за один недельный период |

| **Backfill** | Ретроспективный сбор данных за прошлые периоды |
| **Re-queue** | Повторная постановка в LLM-очередь (данные уже собраны, но LLM не обработан) |
